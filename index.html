2なぜこの設計か（設計判断）

論点決定理由
取り出すCSV会計明細のみ（1つ）会計明細を営業日締めで集計すると日別売上が正規データと完全一致。売上集計CSVは不要
反映先各店の週次レポートシート稜也さんの担当は各店シートまで。集約シートは絵里
実行アカウント責任者（稜也さん）のGoogleアカウント各店シートへのアクセス権を統括の権限で担保。個人PC/トークンに依存させない
週ブロック無ければ自動作成各店に「週ブロック作成」を任せない＝現場の手間を最小化
CSV方式 vs 自動連携まずCSV方式Airレジは公式API無し。自動ログイン取得は2段階認証で不安定。CSVを出す数クリックを挟む方が堅牢
商品名の表記全店共通の装飾除去（店ごとの略記は廃止）「マンゴー／MK」等の店ごとのブレを防ぐ。同じ商品はどの店でも同じ表記
商品の集計単位メニュー名＋種別（サイズ）サイズ違い（1本入/3本入）を分ける。実データと手入力TOP5の一致を検証済み
検証（伊勢佐木店 6/8〜6/14）：日別売上は営業日締め集計で7日とも完全一致、商品別TOP5は売上金額が5位まで完全一致（販売数のみ±1の手集計差）。

3必要なもの（前提）

  • Mac（稜也さんの作業PC）＋ Claude Code 導入済み。
  • 稜也さんのGoogleアカウントで、各店の週次レポートシート（麻布／新宿／伊勢佐木／藤沢）に編集権限があること。
  • 対象店のAirレジから会計明細CSVをエクスポートできること。
権限が無いシートがある場合は絵里に共有を依頼。※確認事項：稜也さんが4店ぶんのAirレジから会計明細を出せる体制か（出せない店は当面その店がCSVだけ用意 → 稜也さんがコマンド実行）。

4セットアップ手順（初回のみ）

4-1. Claude Code の確認

「ターミナル」を開き claude --version でバージョンが出ればOK。

4-2. 一式を配置

絵里から受け取るフォルダ airregi_weekly_setup_guide/ を作業フォルダに置く。中身：SETUP_GUIDE.html（この文書）／build_store_weekly.py（反映スクリプト）／sheet_ids.json（各店シートID）／auth_setup.py（認証）。

4-3. 責任者アカウントでGoogle認証 最重要

  1. python3 auth_setup.py を実行。
  2. ブラウザで稜也さんのGoogleアカウントでログインし、Sheetsの編集を許可。
  3. token.json が保存され、以降は稜也さんの権限で各店シートを読み書きする。
注意：token.json は鍵情報。クラウドにアップロード・他人共有はしない。

4-4. 動作確認（ドライラン）

python3 build_store_weekly.py --store 伊勢佐木 --week 6/8-6/14 --dry-run

--dry-run はシートに書き込まず集計結果だけ表示。妥当なら本番（外して実行）へ。

5毎週の運用（月曜昼まで）

① 会計明細CSVをエクスポート

Airレジ管理画面 → 売上 → 会計明細を対象週（月〜日）でCSV出力。Macの「ダウンロード」フォルダに保存される（操作はこれだけ）。

② 反映コマンドを実行

python3 build_store_weekly.py --store all --week 6/8-6/14

ダウンロードフォルダの会計明細CSVを自動で拾い、対象週の日別売上・TOP5を各店シートに反映。当該週ブロックが無ければ自動で作成してから書き込む。--store 新宿 のように1店ずつでも可。

③ あとは各店が気づき欄だけ記入

数字は入った状態なので、各店長は気づき・改善点・提案だけを書く（火曜昼MTGまで）。

6仕様リファレンス

日別売上（会計明細から）

  • 会計明細CSV（CP932）の取引行から、合計＝売上・人数＝客数を取得。
  • 営業日締め：会計時刻が朝（既定6時）より前の取引は前日に計上（深夜営業の日跨ぎ対応）。これで正規の日別売上と完全一致する。
  • 週計・前週比・客単価・前日比はシート側の数式なので触らない。

商品別TOP5（会計明細から）

  • 商品明細行を メニュー名＋種別1 で集計（売上＝価格×数量／販売数＝数量、価格0の付帯行は除外）→ 売上降順TOP5。週判定も営業日締めで統一。

商品名の表示（全店共通）

  • 集計は正式名のまま／表示だけ装飾除去（【…】・（…）・新基準対応・510規格フルリキッド 等）。例：【新基準対応】510規格フルリキッド　陽キャ（GOODCHILL） → 陽キャ。
重要：集計キーは短縮しない（別カテゴリの同名商品が混ざる）。短縮は表示専用。

7トラブルシュート

症状対処
日別売上が正規データと数日ずれる営業日の締め時刻が店で違う可能性。--cutoff 7 等で締め時刻（時）を調整（既定6時。開店前の時刻なら同結果）
文字化けする会計明細はCP932。スクリプトは自動対応。DLしたCSVをそのまま使う（Excelで保存し直さない）
権限エラー（403等）稜也アカウントに対象シートの編集権が無い。絵里に共有依頼 → python3 auth_setup.py で再認証
TOP5が店の体感とずれるサイズ違いの統合や返品の扱いを確認。販売数±1は手集計差で正常

8引き継ぎ・連絡

  • 仕様・スクリプトの不明点は絵里へ。作業者は稜也さんに集約（増やしすぎず、確実に動かせる状態を保つ）。
  • 新しい店の追加は sheet_ids.json に1行足すだけ。

付録：完全自動（将来）について

Airレジは公式APIが無いため完全自動化は「自動ログイン＋画面操作」になり、2段階認証・セッション維持が不安定（原付市場のLINE公式でも同種の壁で手動運用）。まずCSV方式で運用を固め、頻度・負荷を見てから判断するのが堅実。
